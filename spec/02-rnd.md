# R&D-документ: Lost Ark Raid Tracker

## 1. Краткое описание задачи

Frontend-приложение для учёта пройденных рейдов игроками Lost Ark и полученного золота. Приложение хранит данные локально в браузере (localStorage), поддерживает экспорт/импорт конфигурации.

## 2. Цель системы

Создать удобный инструмент для игроков Lost Ark, позволяющий:
- отслеживать прогресс прохождения рейдов;
- учитывать получаемое золото;
- вести учёт нескольких игровых персонажей;
- хранить настройки между сессиями без серверной части.

---

## 3. Основные пользовательские сценарии

### Сценарий 1: Учёт персонажей
1. Пользователь добавляет нового персонажа (имя, ГС, класс).
2. Пользователь может редактировать данные персонажа.
3. Пользователь может удалить персонажа.

### Сценарий 2: Назначение рейдов персонажу
1. Пользователь выбирает персонажа.
2. Пользователь добавляет рейд к персонажу, выбирая:
   - Название рейда;
   - Сложность (для каждой сложности — свой requiredGearScore, regularGold, limitedGold).
3. Пользователь отмечает рейд как пройденный/непройденный.
4. Пользователь может удалить назначение рейда.

### Сценарий 3: Управление библиотекой рейдов
1. Пользователь создаёт новый рейд с набором сложностей.
2. Пользователь редактирует параметры рейда.
3. Пользователь удаляет рейд (из библиотеки и из всех назначений).

### Сценарий 4: Смена темы
1. Пользователь переключает тему между светлой и тёмной.
2. Настройка сохраняется между сессиями.

### Сценарий 5: Экспорт/Импорт конфигурации
1. Пользователь экспортирует все данные (персонажи, рейды, назначения, настройки) в JSON-файл.
2. Пользователь импортирует ранее сохранённый JSON-файл (полная замена текущих данных).

## 6. Ограничения первой версии

1. **Без авторизации** — все данные локальны, без синхронизации между устройствами.
2. **Без серверной части** — только client-side хранение (localStorage).
3. **До 30 персонажей** — ограничение накладывается на этапе добавления.
4. **Без иконок классов** — поле `characterClass` хранится, иконки не реализуются (заглушка/мок).
5. **Без валидации ГС** — проверка requiredGearScore не блокирует добавление, только предупреждение.
6. **Экспорт/импорт — полная замена** — нет слияния данных, только перезапись.

## 7. Модель данных

### Character
| Поле | Тип | Описание |
|------|-----|----------|
| id | string (UUID) | Уникальный идентификатор |
| name | string | Имя персонажа |
| gearScore | number | Уровень снаряжения (ГС) |
| characterClass | string | Класс персонажа |

### Raid
| Поле | Тип | Описание |
|------|-----|----------|
| id | string (UUID) | Уникальный идентификатор |
| name | string | Название рейда |
| difficulties | Difficulty[] | Массив сложностей |

### Difficulty
| Поле | Тип | Описание |
|------|-----|----------|
| type | enum | Одиночный, Обычный, Героический, Кошмар |
| requiredGearScore | number | Требуемый ГС для входа |
| regularGold | number | Обычное золото |
| limitedGold | number | Лимитированное золото |

### CharacterRaid (назначение)
| Поле | Тип | Описание |
|------|-----|----------|
| id | string (UUID) | Уникальный идентификатор |
| characterId | string | Ссылка на персонажа |
| raidId | string | Ссылка на рейд |
| difficultyType | enum | Выбранная сложность |
| isCompleted | boolean | Прогресс прохождения |
| createdAt | number | Timestamp создания |

### AppSettings
| Поле | Тип | Описание |
|------|-----|----------|
| theme | 'light' \| 'dark' | Текущая тема |

## 10. Риски реализации

| Риск | Вероятность | Влияние | Меры снижения |
|------|-------------|---------|---------------|
| Рост сложности при управлении сложностями рейдов | Средняя | Средняя | Упростить UI: единая форма с динамическим добавлением сложностей |
| Проблемы с localStorage при больших объёмах данных | Низкая | Высокое | Добавить лимит на количество записей, сериализацию в JSON |
| Потеря данных при импорте | Средняя | Высокое | Валидация JSON перед импортом, предупреждение о перезаписи |
| Сложность синхронизации CharacterRaid при удалении | Низкая | Среднее | Cascade delete: при удалении рейда удалять связанные назначения |

## 12. Рекомендации для архитектурного этапа

1. **Single Page Application** — один маршрут (Dashboard), модальные окна для редактирования.
2. **Pinia для state management** — три store: characters, raids, settings.
3. **Реактивная связь** — CharacterRaid хранит только id, computed-свойства для данных.
4. **Ранняя валидация** — Yup или Zod для форм ввода.
5. **Резервное копирование** — перед импортом сохранять текущее состояние в отдельный ключ localStorage.
6. **Компонентный подход** — Atomic Design: atoms → molecules → organisms.
7. **Тема** — CSS Variables для переключения светлой/тёмной темы.
8. **Не усложнять в первой версии** — без аутентификации, без реальных иконок классов, без серверной синхронизации.
